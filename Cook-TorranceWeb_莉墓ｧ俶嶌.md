## 1. 目的

ゲームCGで標準的に使われる Cook-Torrance BRDF（GGXベースのマイクロファセットモデル）を、**中学数学の知識のみを前提**とした読者が、数式の各記号・各項の意味と計算手順を完全に理解できる状態にする。

「完全理解」の定義:
1. 反射方程式と Cook-Torrance の式に出てくる全記号を、自分の言葉で説明できる
2. パラメータ（roughness, metallic, F0, ライト方向など）を変えたとき、数値と見た目がどう変わるかを予測できる
3. シェーダコード（GLSL）と数式の対応が取れる

## 2. 対象読者

- 中学数学（四則演算、簡単な関数、グラフ）は理解している
- 高校数学（三角関数、ベクトル、微積分）は未習または忘れている
- Substance 3D Designer 等の DCC ツールの利用経験があり、roughness / metallic / base color という言葉は見たことがある

## 3. 設計方針

- **数学は必要になった時に必要な分だけ教える**（前置きで数学書を1冊やらせない）
- すべての数式に「日本語での読み下し文」を併記する（例: `n·l` →「面の向きと光の向きがどれだけ揃っているか（0〜1）」）
- すべての数式にインタラクティブなウィジェットを付け、値をいじると結果が即座に変わることを体験させる
- 各章末に理解度チェック（3〜5問のクイズ）を置く
- 最終的にすべての式が1つのシェーダに合流し、総合ビューポートで完結する

## 4. カリキュラム（章構成）

### 第0章 イントロダクション
- なぜ数式で光を計算するのか（フォトリアルの正体は物理シミュレーション）
- Cook-Torrance がゲーム（UE, Unity, Substance）でどう使われているか
- このサイトの学び方・ゴールの提示
- ウィジェット: 完成版ビューポートのプレビュー（「最後にはこれが全部わかる」）

### 第1章 数学の道具箱①: ベクトル
- ベクトル＝「向きと長さ」の矢印。3D空間では (x, y, z) の3つの数
- 足し算・引き算・スカラー倍
- 長さの計算（三平方の定理の3D版。中学の知識から接続）
- 正規化（長さを1にする）＝「向きだけ取り出す」
- CGで使う主要ベクトルの導入: **n**（法線）, **l**（ライト方向）, **v**（視線方向）, **h**（ハーフベクトル）
- ウィジェット:
  - W1-a: 2D/3Dベクトル操作ツール（ドラッグで矢印を動かし、成分と長さがリアルタイム表示）
  - W1-b: 球体上の点をクリックすると n, l, v, h の矢印が表示されるミニビューア

### 第2章 数学の道具箱②: 内積と cosθ
- 角度と cos の直感（単位円は使うが、「横方向の影の長さ」という説明で導入）
- cos のグラフ: 0°→1, 90°→0, それ以上→負
- 内積の定義と計算方法（成分の掛け算の和）
- **内積 = 「2つの向きの揃い具合」**。CGで最頻出の演算
- `max(0, n·l)` がなぜ必要か（裏側は照らされない）
- ウィジェット:
  - W2-a: 2本の矢印をドラッグすると角度・cosθ・内積値が連動表示
  - W2-b: 球体の各点の `n·l` を色で可視化（Lambertシェーディングの原型を先取り）

### 第3章 数学の道具箱③: 「全部足す」＝積分と半球
- 「積分」という言葉は使うが、**定義は「細かく分けて足し合わせること」だけ**。極限・微分は扱わない
- 面の上の半球（その点に届く光は半球のあらゆる方向から来る）
- 立体角 ω の直感（「方向の集まりの広さ」。厳密な定義は補足に隔離）
- ウィジェット:
  - W3-a: 半球を分割数スライダーで細かくし、離散的な足し算が滑らかな積分に近づく様子を可視化
  - W3-b: 半球上の光源（環境光イメージ）の寄与を1方向ずつ足していくアニメーション

### 第4章 反射方程式（レンダリング方程式の反射項）
```
Lo(v) = ∫Ω f(l, v) · Li(l) · (n·l) dωl
```
- 各記号の完全分解:
  - Lo: カメラに向かう光の量（＝画面のピクセルの明るさ）
  - Li: 方向 l から入ってくる光の量
  - f: BRDF（次章）。「この材質は、l から来た光を v 方向へどれだけ返すか」
  - (n·l): 斜めから当たる光は薄まる（第2章の復習）
  - ∫Ω dω: 半球の全方向について足す（第3章の復習）
- ゲームでの簡略化: ポイントライト1個なら積分が消えて
  `Lo = f(l, v) · Lc · (n·l)` になる。**本サイトの実装はこの形を採用**（Li の積分はIBLの話として補足に留める）
- ウィジェット:
  - W4-a: 式の各項をクリックすると該当部分の解説がポップアップ＋ビューポート上で該当要素がハイライトされる「数式分解ビューア」

### 第5章 BRDFとは何か
- 定義の直感: 「入射方向と出射方向のペアごとに反射率を返す関数」
- 材質の違い＝BRDFの違い（チョーク vs 鏡 vs プラスチック）
- diffuse + specular の2成分に分ける発想:
  `f = kd·f_diffuse + f_specular`
- ウィジェット:
  - W5-a: BRDFローブ可視化ツール。入射方向を動かすと、反射の分布（ローブ）が極座標表示で変形。プリセット（完全拡散/光沢/鏡面）切替

### 第6章 ディフューズ成分: Lambert
```
f_diffuse = albedo / π
```
- 表面下で光が散乱して「あらゆる方向へ均等に」出ていくイメージ
- albedo（ベースカラー）＝波長ごとの反射率＝RGBの3つの数
- **なぜ π で割るのか**: エネルギー保存。半球全体に均等に配ると合計が π 倍になるため（導出は補足。本文は「配りすぎ防止の正規化」という直感で通す）
- ウィジェット:
  - W6-a: 球体ビューア（Lambertのみ）。ベースカラー・ライト色・ライト方向を操作可能
  - W6-b: π で割る/割らないの比較トグル（エネルギーが1を超える様子を数値で表示）

### 第7章 スペキュラ成分①: マイクロファセット理論と全体像
```
f_specular = D(h) · F(v,h) · G(l,v,h) / (4 · (n·l) · (n·v))
```
- 表面は微細な鏡の集合体という考え方
- ハーフベクトル h の役割: 「l を v に反射させる向きのマイクロファセットだけが光る」
- D, F, G それぞれの担当の概要（詳細は次の3章）
- 分母 4(n·l)(n·v) は補正項（導出は補足に隔離。「単位合わせ」とだけ説明）
- ウィジェット:
  - W7-a: 凸凹断面図シミュレータ。roughnessスライダーで表面のギザギザが変化し、光線が反射する様子を2Dアニメーション表示

### 第8章 スペキュラ成分②: D項（法線分布関数 / GGX）
```
D_GGX(h) = α² / (π · ((n·h)²·(α²−1) + 1)²)    ただし α = roughness²
```
- 「マイクロファセットの向きが h と揃っている割合」
- roughness とハイライトのサイズ・鋭さの関係
- GGX の特徴（裾が長い＝リアルなハイライトの尾）
- 式は1ステップずつ計算過程を分解して表示（`n·h` → 2乗 → … の順に電卓的に追う）
- ウィジェット:
  - W8-a: D関数のグラフ（横軸: n·h の角度、縦軸: D値）。roughnessスライダー連動
  - W8-b: 球体ビューア（D項のみを白黒表示するモード）

#### コラム: オリジナルの Cook-Torrance は GGX ではない（Beckmann 分布）
- 1982年の Cook & Torrance の原論文では、分布関数 D に **Beckmann 分布**が使われていた:
  ```
  D_Beckmann(h) = exp( ((n·h)² − 1) / (m² · (n·h)²) ) / (π · m² · (n·h)⁴)
  ```
  （m は表面の粗さ。exp は「ネイピア数のべき乗」。中学数学の範囲外なので、「入力が0から離れるほど急激に小さくなる関数」という直感だけ与える）
- 2007年に Walter らが GGX（＝Trowbridge-Reitz, 1975）を再発見・提唱し、現在のゲームCG（UE, Unity, Substance）ではこちらが事実上の標準になった
- **なぜ GGX が勝ったのか**: 裾（tail）が長い。Beckmann はハイライトの周囲が急激に暗くなるが、GGX はぼんやりとした光の尾を残す。実測データや実際の見た目に近いのはこちら
- 「Cook-Torrance モデル」という言葉は、現在では **D·F·G/(4(n·l)(n·v)) というマイクロファセットの枠組み全体**を指し、D に何を入れるかは実装依存、という理解が実務上正しい
- ウィジェット:
  - W8-c: **D関数 比較グラフ**。GGX と Beckmann を同一グラフに重ねて表示。roughness スライダー連動で「裾の長さ」の違いを目視確認
  - W8-d: 球体ビューアの D 表示モードで GGX / Beckmann をトグル切替（見た目の差を確認。本文の実装は GGX に固定）

### 第9章 スペキュラ成分③: F項（フレネル / Schlick近似）
```
F_Schlick = F0 + (1 − F0) · (1 − v·h)⁵
```
- 浅い角度で見ると何でも鏡になる現象（水面・机の実例写真）
- F0（垂直入射時の反射率）: 非金属はほぼ0.04、金属はベースカラーそのもの
- **metallic パラメータの正体はここ**（F0 と diffuse の切替）
- 5乗の意味は「端で急に立ち上がるカーブを作るための近似」でよい
- ウィジェット:
  - W9-a: フレネルカーブのグラフ（横軸: 角度、縦軸: 反射率）。F0スライダー連動
  - W9-b: 球体ビューア（F項のみ表示モード。縁が明るくなるリムを確認）

### 第10章 スペキュラ成分④: G項（幾何減衰 / Smith + Schlick-GGX）
```
G = G1(l) · G1(v),   G1(x) = (n·x) / ((n·x)(1−k) + k),   k = (roughness+1)²/8
```
- 凸凹による「影」（シャドウイング）と「遮蔽」（マスキング）の直感
- 浅い角度＋高roughnessで暗くなる理由
- ウィジェット:
  - W10-a: 2D断面図でシャドウイング/マスキングが起きる光線を赤表示
  - W10-b: G関数のグラフ（角度 × roughness）
  - W10-c: 球体ビューア（G項のみ表示モード）

### 第11章 全体の合成とエネルギー保存
```
f = kd · (albedo/π) + D·F·G / (4(n·l)(n·v))
kd = (1 − F) · (1 − metallic)
Lo = f · Lc · (n·l)
```
- スペキュラに使われた光はディフューズに回らない（kd の意味）
- metallic ワークフロー: metallic=1 で diffuse 消滅、F0=albedo
- 数値例のフルトレース: 具体的な n, l, v, roughness 等を1組決め、**電卓レベルで最終RGB値まで手計算を全ステップ表示**するインタラクティブ計算機
- ウィジェット:
  - W11-a: ステップ実行計算機（各中間値: h, n·l, n·v, n·h, v·h, D, F, G, kd, 最終色 を表形式で表示。入力は全部スライダー）

## 6. 技術スタック（ビルド不要・依存ライブラリゼロ）

**最優先要件**: zip を解凍して `index.html` をダブルクリックするだけで全機能が動くこと（`file://` で完動）。同じファイル群が GitHub Pages でもそのまま動くこと。

この要件から、以下は**すべて禁止**する:
- npm / ビルドステップ（TypeScript のコンパイル、バンドル）
- ES Modules（`<script type="module">`, `import`/`export`）→ `file://` では CORS で読み込み失敗する
- `fetch()` / `XMLHttpRequest` によるローカルファイル読み込み（章のHTML断片、シェーダの .glsl ファイル、JSONデータ等）→ `file://` ではブロックされる
- CDN からの外部読み込み（Three.js, KaTeX 等）→ オフラインで動かなくなる

| 項目 | 選定 | 理由 |
|---|---|---|
| 言語 | 素の HTML / CSS / JavaScript（ES2015、クラシックスクリプト） | ビルド不要。`<script src="...">` は `file://` でも動く |
| 3D | **WebGL を自前で直接呼ぶ**（Three.js 不使用） | 依存ゼロ。球1個＋オービットカメラなら自前実装で十分（約300行）。シェーダ内容が学習対象そのものなので、ライブラリに隠されない方が教材として正しい |
| シェーダ | GLSL を **JS内の文字列として埋め込む**（.glsl ファイルにしない） | `fetch()` を使わないため |
| 2Dグラフ・図 | Canvas 2D 自前描画 | 依存ゼロ |
| 数式表示 | **CSS で組む自前の数式コンポーネント**（KaTeX 不使用） | KaTeX はフォントファイルの読み込みが必要で `file://` だと不安定。分数・べき乗・ギリシャ文字程度なら CSS（flexbox）と Unicode で十分きれいに組める。式の一部をクリック可能にする要件とも相性が良い |
| 状態保存 | なし（メモリ内のみ） | — |
| デプロイ | GitHub Pages（**Actions 不要**、main ブランチの / をそのまま公開） | ビルドがないのでソース＝公開物 |

## 8. 動作確認・公開方法（重要要件）

### 8.1 PC: ダウンロードして開く
1. GitHub の `Code` → `Download ZIP`
2. 解凍
3. `index.html` をダブルクリック → ブラウザで開く。**サーバー不要・ネット接続不要で全機能動作**

### 8.2 iPhone / スマホ: ダウンロードせずリンクをタップして開く（GitHub Pages）
可能。GitHub Pages で公開すれば、iPhone の Safari でURLを開くだけで動く（WebGL・タッチ操作とも対応）。

設定手順（リポジトリ側で一度だけ）:
1. GitHub のリポジトリ → `Settings` → 左メニュー `Pages`
2. `Source` を **Deploy from a branch** に設定
3. Branch を `main` / フォルダを `/ (root)` に設定して Save
4. 数分後、以下のURLで公開される:
   **https://ytaniytani.github.io/Cook-TorranceWeb/**
5. 以降は main に push するたびに自動で反映される（GitHub Actions の設定ファイルは不要）

注意点:
- GitHub Pages はリポジトリが public であることが前提（private リポジトリで Pages を使うには有料プランが必要）
- ブックマークまたはホーム画面に追加しておけば、アプリのように起動できる

### 8.3 補足: GitHub上でファイルを直接開いても表示されない
GitHub のファイルビューでは HTML はソースコードとして表示され、実行されない。必ず 8.1 か 8.2 の方法を使う。

## 9. 非機能要件

- 日本語UI
- **モバイル対応**（iPhone Safari を確認対象に含める）:
  - レイアウトは幅に応じて1カラムに折返し
  - 3Dビューポートはタッチ操作対応（1本指ドラッグ＝回転、2本指ピンチ＝ズーム）
  - ページのスクロールと3D操作のジェスチャが競合しないよう、キャンバス上では `touch-action: none` を指定
  - スライダーはタッチでも掴みやすいサイズにする
- 外部APIなし・外部CDNなし・完全オフライン動作
- 目標: リポジトリ全体で 1MB 以下（依存ライブラリがないため容易にクリア）

## 10. 開発フェーズ

| フェーズ | 内容 |
|---|---|
| P1 | 総合ビューポートのコア実装（シェーダ・カメラ・パラメータUI・項別表示） |
| P2 | 第0〜4章（数学基礎＋反射方程式）＋対応ウィジェット |
| P3 | 第5〜11章（BRDF本体）＋対応ウィジェット＋ステップ計算機 |
| P4 | 第12章・クイズ・補足・仕上げ・GitHub Pages デプロイ |

シェーダとステップ計算機は同一ロジックの二重実装になるため、P1でGLSLを確定させてからP3でCPU側へ移植する。

## 11. スコープ外（v1では扱わない）

- IBL / 環境マップ / HDRI
- 影（シャドウマッピング）
- 異方性、クリアコート、SSS などの拡張BRDF
- **テクスチャマップ（albedo map / roughness map / normal map 等）の読み込み・適用**
  - ただし roughness・metallic・base color は「1枚の値」としてスライダー／カラーピッカーで自由に変更できる（第5.4節）。テクスチャで空間的に変化させることをしない、という意味
- マルチライト

## 12. 確定した実装構成

| 項目 | 採用 |
|---|---|
| D（法線分布） | **GGX / Trowbridge-Reitz**（α = roughness²） |
| F（フレネル） | Schlick 近似 |
| G（幾何減衰） | Smith + Schlick-GGX（k = (roughness+1)²/8） |
| Diffuse | Lambert（albedo/π） |
| ライト | 1灯（方向 or 点光源）、解析的評価 |
| Beckmann 分布 | 第8章コラムで解説＋比較ウィジェット（W8-c / W8-d）として実装 |
| テクスチャ | 不使用。roughness / metallic / base color はスライダー・カラーピッカーで操作 |
| 実装形態 | 素の HTML/CSS/JS。ビルド不要、依存ライブラリゼロ、WebGL 自前実装 |
| 確認方法 | ①zip解凍→index.html ダブルクリック（オフライン可） ②GitHub Pages の URL をタップ（iPhone可） |
